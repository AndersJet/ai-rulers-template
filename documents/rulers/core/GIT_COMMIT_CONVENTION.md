> **角色**: 面向 AI 辅助变更的通用提交门禁与约定式提交指南。

> **优先级声明**：本文件的提交格式规则（中文 subject/body）覆盖一切 skill、command 和 workflow 中的 commit 步骤。任何指令源的 commit 操作不得绕过本文件的格式检查和 CHANGELOG 门禁。

## 1. 通用提交门禁

提交前，AI 必须：

1. 审阅工作区与暂存区差异（`git status`，`git diff`，`git diff --cached`）。
2. 只暂存与已批准任务相关的文件，避免混入密钥、生成产物或无关清理。
3. **主动生成**提交信息草案：分析 `git diff --cached`（含 `--stat` 和各文件摘要），推断 type、scope、subject 和 body。
    - **禁止**向用户询问"提交信息写什么"——这是 AI 的职责。
    3.5 运行 `git log --oneline -20` 了解仓库历史提交格式。注意：历史中可能存在不符合本规范的 commit（如英文 subject）。这些提交是在本规范生效前产生的，**不得作为格式参考**。实际提交格式必须以本文件第2-5节为准，不得模仿历史中的违规格式。
4. 展示文件清单与提交信息草案，等待用户确认后执行。
    - **例外**：用户已明确说"提交/ commit 已暂存内容"时，跳过反复确认，仅就提交信息草案等待一次确认即可。
5. **CHANGELOG 同步检查**
    - 提交前必须依据 `CHANGELOG_MAINTENANCE.md` 判断本次变更是否需要记录。
    - 若需要记录，必须在 `[Unreleased]` 下追加条目，并与代码变更一起 `git add`。
    - CHANGELOG 更新必须与代码变更进入同一个 commit，禁止事后单独提交。

如果用户没有要求提交，则不要提交。

### 1.1 未暂存更改保护

处理提交、暂存或清理工作区时，AI 必须保护未暂存更改：

- **绝对禁止**在未获得用户明确批准前，暂存、提交、丢弃、还原、覆盖或格式化未暂存更改。
- 未暂存更改必须先按文件识别归属：本任务相关、用户/其他 agent 并行变更、或未知来源。
- 只允许暂存已确认属于本任务且已获批准进入本次提交的更改；对同一文件内混合变更，必须使用最小范围暂存，无法安全拆分时先询问用户。
- 对用户/其他 agent 的并行变更或未知来源更改，只能报告现状并请求裁决，不得自行清理或纳入提交。
- 即使用户要求"提交当前改动"，也不得默认包含未暂存更改；必须明确说明将提交的是暂存区，还是经确认后的指定未暂存文件。

## 2. 提交格式

### 2.1 约定式提交格式

```text
<type>[optional scope]: <subject>
```

推荐类型：

- `feat`：面向用户或领域的新功能。
- `fix`：缺陷修复。
- `docs`：仅文档变更。
- `refactor`：不改变行为的结构调整。
- `test`：新增或修改测试。
- `chore`：不影响运行时行为的维护（依赖升级、构建脚本、CI 配置、工具链等）。

---

## 3. 提交信息指南

- `type` 保持英文。
- `subject` 使用中文。
- subject 使用动宾结构，不以中文句号结尾。
- body 使用中文，且**必须**包含以下要素（每条以 `-` 开头，直接罗列）：
    1. 改了什么：按文件或模块分组，列出核心变更点。
    2. 为什么：说明本次变更的动机或解决的问题。
    3. 影响范围（如有）：若变更涉及接口、数据库、配置等，须明确标注。
- 当作用域有助于识别受影响区域时使用作用域。
- 适用时包含破坏性变更说明或迁移影响。

示例：

```text
feat(auth): 新增账户恢复流程

- 新增 /api/v1/auth/recover 接口，支持邮箱+短信双通道验证
- 解决用户忘记密码后无法自助找回的问题
- 影响范围：auth 模块、用户表新增 recovery_token 字段
```

```text
fix(settings): 保留通知偏好设置

- 修复设置页刷新后通知开关状态丢失
- 原因为 localStorage 键名与读取逻辑不一致
```

```text
refactor(prompts): 精简个人任务查询提示模板

- 将 SYS_PERSONAL_TASK_*_QUERY 从冗长多步骤描述压缩为精简流程，减少 token 消耗
- 知识库同步新增业务编号+目标属性边界样本 140 条
```

---

## 4. 安全提醒

- 除非用户明确请求并确认，否则绝不修订提交、变基、重置、强制推送、删除分支或执行破坏性仓库操作。
- 除非用户明确请求并确认，否则绝不丢弃、还原、覆盖、清理或重格式化未暂存更改。
- 除非用户明确请求并确认，且记录原因，否则绝不绕过钩子或验证门禁。
- 如果钩子或验证失败，停止、报告失败，并在尝试新提交前修复根本问题。

## 5. 提交信息生成指引（AI 执行标准）

AI 生成提交信息时，必须按以下顺序推理：

### 5.1 收集证据

1. 运行 `git diff --cached --stat` 获取变更概况（文件数、增减行数）。
2. 运行 `git diff --cached -- <file>` 获取各文件的具体差异摘要（取前 30~50 行即可，不必全读）。
3. 运行 `git diff --cached -- CHANGELOG.md` 确认 CHANGELOG 已同步更新。

### 5.2 推断 type

| 变更特征 | 推荐 type |
|---|---|
| 新增功能、接口、页面、流程 | `feat` |
| 修复 bug、异常处理 | `fix` |
| 仅文档、注释、README、知识库 | `docs` |
| 代码重构、结构调整、提示模板精简，**不**改变外部行为 | `refactor` |
| 新增/修改测试 | `test` |
| 依赖升级、构建脚本、CI 配置、工具链 | `chore` |

- 单条 commit 混合多种性质变更时，取**占比最高**的类型。
- 纯结构调整、提示模板压缩且无行为变更，**必须**使用 `refactor`，而非 `feat` 或 `fix`。

### 5.3 推断 scope

- 取受影响文件的**最上层公共目录**或**核心模块名**。
- 单模块内多文件变更 → scope 为该模块名（如 `prompts`、`auth`）。
- 跨模块变更 → 使用共同父目录或省略 scope。
- 纯文档变更 → scope 可省略或使用 `docs`。

### 5.4 构造 subject

- 中文动宾结构，不超过 50 字。
- 精确描述**做了什么**，而非**为什么做**。
- 禁止以"修复了…""优化了…"等模糊动词开头；应写具体动作，如"精简…""新增…""调整…"。

### 5.5 构造 body

按以下要素组织（每条以 `-` 开头，直接罗列）：

1. 改了什么：按模块/文件分组，列出核心变更点。
2. 为什么：说明动机（如"减少 token 消耗""修复状态丢失""提升边界覆盖"）。
3. 影响范围（如适用）：标注接口、字段、配置、数据库等外部可见变化。

示例：

```text
refactor(prompts): 精简个人任务查询提示模板并扩充意图识别边界用例

- 精简 SYS_PERSONAL_TASK_*_QUERY 五个提示模板，移除冗余步骤描述
- 调整告警查询/恢复提示的 call_indicator_query 参数，限定返回字段
- 知识库新增业务编号+目标属性边界样本 140 条
- kb_test 重构为轻量级导入，P0 混淆边界扩展至 12 组
- 减少提示模板 token 消耗，提升意图识别边界覆盖
```

### 5.6 最终检查清单

在将提交信息展示给用户前，AI 必须自检：

- [ ] type 是否与变更特征匹配？
- [ ] scope 是否准确反映受影响区域？
- [ ] subject 是否为动宾结构且不超过 50 字？
- [ ] body 是否说明了"改了什么"和"为什么"？
- [ ] 提交信息中是否未出现密钥、Token、密码或内部 URL？
- [ ] 是否已检查未暂存更改，且未擅自暂存、提交或丢弃任何未获批准的更改？
- [ ] 本次 `feat` / `fix` 是否已在 `CHANGELOG.md` 的 `[Unreleased]` 下追加条目？
- [ ] `CHANGELOG.md` 是否与代码变更一起暂存并纳入同一 commit？
