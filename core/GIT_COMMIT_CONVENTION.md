# Git 提交规范

> **角色**: 面向 AI 辅助变更的通用提交门禁与约定式提交指南。

```yaml
metadata:
  applies_to:
    - .git/**
    - "**/*"
  trigger_keywords:
    - git
    - commit
    - conventional commits
    - staged diff
    - working tree
    - 提交
    - 提交规范
    - 约定式提交
    - 暂存区差异
    - 工作区
    - 提交门禁
  must_load_with:
    - ../AGENTS.md
    - HARD_CONSTRAINTS.md
```

---

## 1. 通用提交门禁

提交前，AI 必须：

1. 审阅工作区状态。
2. 审阅未暂存差异。
3. 审阅暂存区差异。
4. 只暂存与已批准任务相关的文件。
5. 避免暂存密钥、生成的本地产物、无关人工工作或任务范围外的大范围清理。
6. 提供拟提交文件的简明摘要，并说明它们为什么属于本任务。
7. 提供拟使用的提交信息。
8. 在运行提交命令前等待明确人工确认。

如果用户没有要求提交，则不要提交。

---

## 2. 约定式提交格式

使用此格式：

```text
<type>[optional scope]: <subject>
```

推荐类型：

- `feat` 表示面向用户或领域的新功能。
- `fix` 表示缺陷修复。
- `docs` 表示仅文档变更。
- `refactor` 表示不改变行为的结构调整。
- `test` 表示新增或修改测试。
- `chore` 表示不影响运行时行为的维护。
- `build` 表示构建系统或依赖变更。
- `ci` 表示持续集成配置变更。

---

## 3. 提交信息指南

- `type` 保持英文。
- `subject` 使用中文。
- subject 使用动宾结构。
- subject 不以中文句号结尾。
- body 使用中文说明“改了什么”和“为什么”。
- 当作用域有助于识别受影响区域时使用作用域。
- 适用时包含破坏性变更说明或迁移影响。

示例：

```text
feat(auth): 新增账户恢复流程
fix(settings): 保留通知偏好设置
docs(rulers): 新增可复用核心规则模板
refactor(payments): 隔离支付适配器边界
test(search): 覆盖空查询处理
```

---

## 4. 安全提醒

- 除非用户明确请求并确认，否则绝不修订提交、变基、重置、强制推送、删除分支或执行破坏性仓库操作。
- 除非用户明确请求并确认，且记录原因，否则绝不绕过钩子或验证门禁。
- 如果钩子或验证失败，停止、报告失败，并在尝试新提交前修复根本问题。
